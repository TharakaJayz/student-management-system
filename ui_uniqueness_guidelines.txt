9) ACCESSIBILITY + QUALITY RULES
==================================================

- visible focus rings
- keyboard navigable controls

Readability rules:
- avoid tiny text

- keep line lengths manageable

Usability guardrails:
- avoid clutter
- avoid too many colors in one section
- keep primary action obvious

==================================================
10) DO / DON'T LIST (Strongly Recommended)
==================================================
DO
- Keep consistent spacing and radius across all pages
- Keep state colors consistent (success/warning/error)
- Keep one visual hierarchy pattern for all cards/tables
- Prefer clarity over decorative elements

DON'T
- No random gradients unless requested
- No glassmorphism unless requested
- No neon palette unless requested
- No mixed button styles in same section
- No overly heavy shadows
- No copying reference 1:1

Recommended similarity range:
- 50-70% (inspired, not cloned)

==================================================
12) BACKEND-TO-UI MAPPING INSTRUCTIONS
==================================================
Always include:
- Data type names (example: ClassRoom, Subject, Class)
- Which fields must be visible in list view
- Which fields appear in create/edit form
- Which fields are hidden/internal
- Which fields are required
- Expected status enums and badge colors

Example:
- Subject:
  - list: name, medium, isActive
  - form: name, medium
- ClassRoom:
  - list: name, location, capacity, isAirConditioned
  - form: name, location, capacity, isAirConditioned

==================================================
13) IMPLEMENTATION CONSTRAINTS (for AI output)
==================================================
Specify upfront:
- Tech stack (HTML/CSS, React, Tailwind, etc.)
- Component framework (if any)
- Icon library preference
- Styling method (CSS modules / styled-components / plain CSS)
- Route/file naming convention
- Reusability requirement level
- Whether mock/static data is acceptable
